# 提案阶段 (Proposal Phase)

> 在开始实现之前，明确要做什么、为什么做、影响范围。

## 触发条件

当任务涉及以下内容时，需要先创建提案：

- ✅ 新功能或新能力（新增 Provider 插件、新 Exporter、GUI 模块）
- ✅ 破坏性变更（Pydantic schema 变更、CLI 参数格式、Store 表结构）
- ✅ 架构调整（Orchestrator 流程变更、新增 Pipeline 阶段）
- ✅ Gate 规则变更（评分阈值、停机条件）
- ✅ 安全模式变更

以下情况可跳过提案：

- ❌ Bug 修复（恢复预期行为）
- ❌ 代码重构（不改变外部行为）
- ❌ 文档更新
- ❌ 非破坏性依赖更新

## 输入

- 任务描述
- 相关上下文：`context/business/DOMAIN.md`、`context/tech/ARCHITECTURE.md`

## 执行步骤

### 1. 检查现有提案

```
是否已有相关提案？
├─ 是 → 阅读提案，进入下一阶段
└─ 否 → 创建新提案
```

### 2. 创建提案文档

提案应包含：

```markdown
# 提案：[简短描述]

## 背景
为什么需要这个变更？

## 目标
这个变更要达成什么？

## 方案
打算怎么做？

## 影响
- 影响的模块：[Orchestrator / Store / Gate / Exporter / Provider / GUI]
- 破坏性变更：[是/否，如是则说明对 CLI、schema、API 的影响]
- 依赖变更：[是/否，如是则说明]
- 对 Draft→Review→Edit→Gate 闭环的影响：[是/否]

## 任务清单
- [ ] 任务 1
- [ ] 任务 2
- [ ] 任务 3
```

### 3. 审核提案

提案需要回答：

- [ ] 问题定义清晰吗？
- [ ] 方案可行吗？
- [ ] 影响范围评估准确吗？
- [ ] 是否影响 CR 闭环或 Gate 规则？
- [ ] 是否需要 Pydantic schema 变更？
- [ ] 任务拆分合理吗？

## 输出

- 提案文档（proposal.md 或等效文档）
- 任务清单
- 【可选】设计文档（如需要）
